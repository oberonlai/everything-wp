# Make Block Command

This command takes a **reference website URL and/or design mockup images**
and automatically produces a consistent block theme design system:

1. Extract design tokens and a component inventory from the live site or mockups
2. Normalize tokens into `theme.json` (and scaffold tooling via `/init-theme`)
3. Implement every component as a **dynamic block** (block.json + render.php + edit.js)
4. Generate a **UI library page** on the local WordPress site containing every
   block in every variant, and hand the URL to the user for review

After the user approves the UI library page, page **assembly** happens
conversationally — the user directs which sections pull posts from the
database, which fields are editable in the admin, etc. See the skill's
Assembly Mode Guide.

## The Constitution — Single Source of Truth

Every visual value has exactly ONE source. Pages may only reference, never
redefine. This table is law for every file generated by this command:

| Visual concern | Single source | Forbidden in pages/templates |
|----------------|---------------|------------------------------|
| Colors / font sizes / spacing | `theme.json` tokens | Hard-coded hex / px values |
| Component markup & styles | Dynamic block + its CSS | Copied markup, class overrides |
| Pages | Templates (reference only) | Inline styles, redefining components |

## Prerequisites (check before starting)

1. **A running local WordPress site** — wp-env, DDEV, or Local. Detect via
   `.wp-env.json` / `.ddev/config.yaml` / asking the user. If none exists,
   offer to set up wp-env (`npm -g @wordpress/env` + a minimal `.wp-env.json`
   mapping the current directory as a theme).
2. **Node.js + npm** — required for `@wordpress/scripts` block builds.
3. **agent-browser** — used for URL extraction (globally installed per user
   setup). Not needed for image-only input; ImageMagick (`magick`) is
   recommended there for palette sampling.
4. The current directory passes the same Greenfield/Augment safety checks as
   `/init-theme` (see that command's Step 0).

## Usage

```
/make-block <reference-url> [additional-urls...]
/make-block <path/to/mockup.png> [more-images...]
/make-block <url> <mockup.png> …            # mixed input is fine
```

Accepted inputs, in any combination:
- **URLs** — extracted live via agent-browser (computed styles = precise tokens)
- **Image files** — design mockups (png/jpg/webp/pdf), local paths or images
  pasted into the conversation

If no input is given, ask for one. For URL-only input, ONE url is enough —
Stage 1 discovers the rest of the site via its sitemap and picks one
representative page per template. For image input, encourage 2-4
representative screens.

**Curated mode (screenshots + URL together)** — the user's screenshots ARE
the page/state curation: **skip sitemap discovery entirely**. Scan only the
exact URL(s) given, solely to calibrate precise token values (computed
styles); the screenshots drive coverage — components, states (hover, open
menus, errors), and breakpoints. URL-derived values win on token conflicts;
images fill everything else.

## Automated Pipeline (no user gates until the end)

### Stage 1 — Extract

**URL input** — do NOT scan only the URL the user gave. Discover the site's
page inventory first, then scan one representative page per template.

#### 1a. Discover pages via sitemap

> **Skip this step in curated mode** (user supplied screenshots alongside the
> URL) — go straight to 1b with only the given URL(s).

```bash
node @everything-wp/skills/wp-block-theme-pipeline/scripts/discover-pages.mjs <site-url>
```

The script reads `robots.txt` + `sitemap.xml` / `wp-sitemap.xml` (recursing
into sitemap indexes), then:
- **Classifies** every URL: `home` / `page` (static) / `archive` (list) /
  `single` (post/product detail) — by URL pattern shape
- **Clusters same-template pages** (e.g. 1,992 × `/blog/{slug}` → ONE
  cluster) so no two pages sharing a layout are ever both scanned
- Outputs a `scanList`: home + every static page + one sample per
  archive/single cluster

If no sitemap exists (exit 2), fall back to crawling the homepage nav links
and classify those URLs manually with the same taxonomy. If the scanList
exceeds ~10 pages, show it to the user and let them trim; otherwise proceed.
Record the full cluster report in `design/page-inventory.json`.

#### 1b. Scan each page in the scanList

```bash
agent-browser open <url>
agent-browser eval "$(cat @everything-wp/skills/wp-block-theme-pipeline/scripts/extract-tokens.js)"
agent-browser screenshot page-<n>.png
```

#### 1c. Layered component extraction — shared → page → atomic

Build the component inventory in THREE passes, in this order:

1. **Shared elements** (appear on every scanned page): header, nav, footer,
   breadcrumbs. Diff the screenshots/DOM across pages — anything identical
   everywhere is extracted ONCE as a shared component and excluded from later
   passes.
2. **Page-level sections** (unique to a page type): hero, card-grid, pricing,
   faq, archive list row, single-post byline. Tag each with the page type it
   came from (`page` / `archive` / `single`).
3. **Atomic elements** (used inside sections): button, tag, avatar,
   section-head — down to **form controls** (text input, select, textarea,
   checkbox/radio, submit button, validation/error states).

This order prevents double-extraction: a button inside the hero is recorded
once as an atom, and the hero references it.

**Image input** — Read each mockup and extract tokens visually:

1. Sample the color palette. When ImageMagick is available, ground your
   estimates with a quantized histogram:
   ```bash
   magick <mockup.png> -format %c -colors 16 histogram:info:- | sort -rn
   ```
2. Estimate the type scale from relative heading/body sizes; if the user has
   the design source (Figma …), ask for exact values instead of guessing.
3. Estimate spacing/radius/container width from measured pixel proportions
   (state the assumed design width, e.g. 1440px).
4. Mark every image-derived value with `"source": "image-estimate"` in
   `design-tokens.raw.json` — the UI library gate is where estimates get
   corrected against the user's eye.

With mixed input, URL-derived values win on conflicts; images fill the gaps
(unseen components, states, layouts).

The component inventory taxonomy (applies to both routes, built via the
three-pass order in 1c):
- **shared** (site-wide): header, nav, footer, breadcrumbs
- **element** (atoms): button, tag, avatar, input, form controls, section-head, …
- **section** (molecules): hero, faq, pricing, card-grid, archive list row, …
- **template** (pages): home, about, archive, single, …

Write to disk (as records, not gates):
- `design/page-inventory.json` — discovered clusters + scanned samples (URL route)
- `design/design-tokens.raw.json` — raw extracted values
- `design/component-inventory.md` — each component with pass level
  (shared/element/section), variants, page type, and the pages it appears on

### Stage 2 — theme.json + tooling

1. Run the `/init-theme` flow in **block** mode (Greenfield or Augment as
   detected) to scaffold `style.css`, base templates, PHPCS/PHPStan/PHPUnit,
   CI, and the build script. Derive the theme name from the directory unless
   the user said otherwise; do NOT re-ask questions answerable from context.
2. Normalize raw tokens into semantic scales and write them into `theme.json`:
   - Colors: cluster similar values, name by role (`base`, `contrast`,
     `primary`, `secondary`, `accent`, `border`, …) — target 6-10 semantic colors
   - Font sizes: snap to a scale (`small`/`medium`/`large`/`x-large`/…)
   - Spacing: derive a spacing scale for `settings.spacing.spacingSizes`
   - Layout: `contentSize` / `wideSize` from the measured container widths
3. Write `design/token-mapping.md` — raw value → semantic name → where it was
   seen, so the user can audit the normalization later.

### Stage 3 — Dynamic Blocks

For every **element** and **section** in the inventory, generate a block under
`blocks/<slug>/` from the skill's block templates:

- `block.json` — attributes = the component's variants and content parameters
- `render.php` — server-side render; markup uses BEM classes, all visual
  values via `var(--wp--preset--*)` tokens
- `edit.js` — editor UI with `@wordpress/components` controls for variants
- `style.css` — the component's CSS, token-references only

Register all blocks from `functions.php` (loop over `blocks/*/block.json` with
`register_block_type`). Set up `package.json` with `@wordpress/scripts` and run
`npm install && npm run build`. Fix build errors before proceeding.

**After each block completes**, immediately add it to `patterns/ui-library.php`
(menu item + anchored variant section — see Stage 4 for the fixed layout).
The UI library grows with the work instead of being assembled at the end.

**Hard rules** (enforced by the Stage 5 audit):
- No hex colors or raw px values in block CSS — tokens only
- No duplicated markup between blocks — shared structure becomes its own block
- Every i18n string uses the theme text domain

### Stage 4 — UI Library Page

1. Generate `patterns/ui-library.php` from the skill's
   `templates/ui-library.php.template`. The layout contract is FIXED:
   - **Left column (25%)**: sticky menu (`position: sticky`), grouped
     Shared / Elements / Sections, one anchor link per block
   - **Right column (75%)**: one anchored section per block — heading
     (anchor id = block slug), then every variant as a labeled live instance,
     separator between blocks
   - **Regeneration rule**: rebuild this pattern after EVERY completed block
     during Stage 3 — menu item + section added together, ordered
     shared → element → section. The pattern is always in sync with
     `blocks/`; a block without a UI library entry is an audit failure.
2. Create a draft page on the local site and give the user both URLs:
   ```bash
   wp post create --post_type=page --post_title='UI Library' --post_status=draft \
     --post_content="$(<the pattern's block markup>)"
   ```
   (Run inside the container for wp-env/DDEV: `wp-env run cli wp …` / `ddev wp …`.)
3. Run the consistency audit (Stage 5) and fix any findings.
4. **STOP and report**: the front-end preview URL, the editor URL, and a
   summary table of generated blocks. This is the single review gate.

### Stage 5 — Consistency Audit (run before reporting, and after any fix round)

```bash
bash @everything-wp/skills/wp-block-theme-pipeline/scripts/audit-consistency.sh
```

- Hard-coded hex/px in block CSS or templates → must be 0
- Duplicated markup structures across blocks/templates → must be 0
- Inventory components with no corresponding block → listed
- Then `composer phpcs && composer phpstan`

Write `design/consistency-report.md`.

## Review Loop

The user reviews the UI library page and reports issues conversationally:
- "主色太暗" → change ONE value in `theme.json`; everything updates
- "button hover 不對" → change that block's CSS; every usage updates
- "少了一個元件" → add the block, regenerate the UI library pattern and page

After each fix round: rebuild (`npm run build`), refresh the page, re-run the
audit. Never fix a visual issue by adding styles anywhere except the single
source.

## Assembly Mode (after UI library approval)

Assembly is conversational, not automated. The user stacks pages by directing
you; follow the decision table in the skill's Assembly Mode Guide:

| User intent | Implementation |
|-------------|----------------|
| "這區塊列出最新 N 篇文章" | Block `render.php` runs `WP_Query`; attributes: `postType`, `count`, `taxonomy` filters |
| "這段文字/按鈕要能在後台改" | Block attributes — edited per-instance in the editor |
| "全站共用設定（電話、社群連結…）" | Generate an options page via the `option-page` skill; blocks read `get_option()` |
| "文章要多一個欄位並顯示在元件上" | `register_post_meta` + editor panel; block reads the meta in `render.php` |

Each assembled page lands as `templates/<slug>.html` (or a page created in the
editor, per user preference) — composed ONLY of existing blocks and core
blocks. Re-run the consistency audit after each page.

## Outputs

```
design/
├── page-inventory.json        # Sitemap discovery: clusters + scanned samples.
├── design-tokens.raw.json     # Extraction record.
├── token-mapping.md           # Raw → semantic audit trail.
├── component-inventory.md     # Component list with variants.
└── consistency-report.md      # Audit results.
theme.json                     # Token single source.
blocks/<slug>/                 # One dynamic block per component.
patterns/ui-library.php        # Review page pattern.
templates/*.html               # Assembled pages (assembly mode).
package.json + build/          # @wordpress/scripts toolchain.
```
